# Почему Expo, а не Bare React Native?

С версии **React Native 0.74 (май 2024)** Expo считается рекомендуемым способом работы с React Native.  
Я умею работать и с bare React Native и чаще использовал именно его, но на текущий момент **Expo** даёт больше преимуществ для демонстрационного проекта и ускоряет разработку и поддержку.

Многие современные проекты используют хотя бы **Expo Modules**, так как это battle-tested решения для рутинных задач с хорошей поддержкой.

Важно понимать, что Expo имеет несколько форматов:

* **Expo Go** — полностью управляемая платформа, функционал ограничен, нет возможности интегрировать нативные модули.
* **Expo Modules** — bare React Native с возможностью подключать модули Expo.
* **Expo Dev Client** — комбинированное решение: во время разработки вы получаете dev kit как в Expo Go, но можете сделать prebuild в обычное bare React Native приложение.

Именно **Expo Dev Client** был выбран для разработки этого приложения.

Если утрировать, процесс prebuild в bare React Native приложение называется **CNG (Continuous Native Generation)**.  
При prebuild нативные папки можно рассматривать как артефакт сборки (аналог `dist/public` в веб-проектах), а конфигурацию Expo (`app.json`) — как своеобразный `dockerfile`.

Это упрощает дальнейшую конфигурацию проекта у других разработчиков и CI/CD, а также настройку **white label** приложений.  

Однако такой подход требует дополнительного внимания к модулям, которые не поддерживают prebuild «из коробки».  
Например, **Яндекс.Карты** пришлось адаптировать вручную через `expo-config-plugins`.

В итоге проект использует **Expo Dev Client** как основную среду разработки с последующей поставкой **bare React Native** приложения.
